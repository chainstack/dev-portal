This page is both a migration guide and an executable runbook for AI coding agents. Paste the one-liner below into any AI coding agent ŌĆö Claude Code, Codex CLI, Cursor, Windsurf, Gemini CLI, Antigravity ŌĆö and the agent will fetch this page and follow it.

## Quick start

Paste this into your AI agent:

```text
get docs.chainstack.com/docs/migrate-to-chainstack-with-ai.md
```

The agent fetches this page and walks through the migration in three phases: Assess ŌåÆ Plan ŌåÆ Implement. It stops after Plan and waits for your approval before touching the codebase.

## Tools used

The runbook calls the [Chainstack MCP server](/docs/chainstack-mcp-server) ŌĆö a single remote server that covers docs search, deployment options, node and project management, platform status, faucets, and pricing. Two URLs, two different purposes:

- [mcp.chainstack.com](https://mcp.chainstack.com) ŌĆö the agent onboarding page. Point any AI agent at this URL and it reads setup instructions, tool reference, and install options.
- `https://mcp.chainstack.com/mcp` ŌĆö the actual JSON-RPC 2.0 endpoint the agent calls once it's set up.

<Note>
  You don't have to register the MCP server with your LLM client. The server accepts JSON-RPC 2.0 over plain HTTP, so the agent can call it directly ŌĆö on demand, no persistent connection, no forced tool list in the context window. If you prefer a lighter install, the server ships a skill file at [mcp.chainstack.com/skill](https://mcp.chainstack.com/skill) that drops into `~/.claude/skills/chainstack/` (or the equivalent for your agent) and triggers HTTP calls only when relevant. Formal MCP registration works too ŌĆö pick whichever fits your workflow.
</Note>

The Assess phase mostly runs on no-auth tools (`search_docs`, `get_doc_page`, `get_chainstack_pricing`, `contact_chainstack`). The `get_deployment_options`, Dry run, and Implement steps require a Chainstack [API key](https://console.chainstack.com/user/settings/api-keys).

<Tip>
  **How to read this page.** When the runbook links to another Chainstack doc page (for example, `/docs/dedicated-node`), call `get_doc_page` to fetch it ŌĆö don't follow the relative link directly.
</Tip>

## Agent instructions

Migrate this project's blockchain RPC endpoints to Chainstack.

### Assess

#### Coverage check

1. Scan the codebase for all blockchain RPC endpoint URLs that are not Chainstack. Check source code, environment variables, config files, and hardcoded URLs.
2. For each endpoint found, identify:
   - The blockchain protocol and network (mainnet/testnet).
   - The RPC methods used (standard, debug/trace, archive queries like historical block lookups).
   - Whether WebSocket, gRPC, or Solana Geyser is needed.
3. Check what Chainstack supports. Do not use web search for Chainstack docs ŌĆö call the MCP server directly:
   - `search_docs` ŌĆö search for each protocol found in the codebase to check supported networks, node types, available API methods, and features like debug/trace or archive.
   - `get_doc_page` ŌĆö get the full doc page for details on a specific chain.
   - `get_deployment_options` (API key required) ŌĆö all supported chains, networks, node tiers, and features (full, archive, debug\_trace).
4. Report what's fully covered, what needs a specific node tier, what's unsupported, and any endpoints that are already dead or deprecated.

<Warning>
  Do not rely on training data for what Chainstack supports. Always verify against the Chainstack docs and `get_deployment_options`. Docs are the primary source of truth ŌĆö they cover features, methods, node configurations, and console-only capabilities. `get_deployment_options` supplements docs with exact deployable chains, tiers, and features. If something can't be verified through either, flag it as uncertain and direct the user to [Chainstack support](https://support.chainstack.com/).
</Warning>

For pricing questions, call `get_chainstack_pricing`. For unsupported chains that may be available as dedicated deployments, sales inquiries, or anything else that needs a Chainstack human, call `contact_chainstack` ŌĆö it works for any user, including those who don't use Telegram.

Flag the following if the codebase needs them ŌĆö they require manual setup in the Chainstack console and are not available through the API:

- [Dedicated Node](/docs/dedicated-node) deployment ŌĆö call `contact_chainstack` to reach the team. [@ChainstackGrowthBot](https://t.me/ChainstackGrowthBot) on Telegram is an alternative for users who prefer it.
- Solana [Yellowstone gRPC Geyser plugin](/docs/yellowstone-grpc-geyser-plugin) ŌĆö node **Add-ons** tab.
- [Unlimited Node](/docs/unlimited-node-add-on) add-on for high throughput ŌĆö node **Add-ons** tab.
- [MEV protection](/docs/mev-protection) ŌĆö node details page.
- [Warp transactions](/docs/warp-transactions) ŌĆö node details page.
- [IP allowlisting and access rules](/docs/access-rules) ŌĆö node **Security** tab.

Answer any follow-up questions ŌĆö use the MCP server tools to provide accurate, specific answers.

#### Cost and savings estimate

If the user is migrating from another RPC provider ŌĆö **QuickNode**, **Alchemy**, or any other ŌĆö estimate their Chainstack cost and the savings versus their current bill.

5. Get the user's current usage from the source provider ŌĆö pick the lowest-friction path:
   - **QuickNode** ŌĆö the Admin API exposes usage directly. Ask the user for a QuickNode API key (created at [dashboard.quicknode.com/api-keys](https://dashboard.quicknode.com/api-keys)) and fetch their billing-period usage:
     ```bash
     curl -s 'https://api.quicknode.com/v0/usage/rpc' \
          -H 'accept: application/json' -H 'x-api-key: <QUICKNODE_API_KEY>'
     ```
     This returns API Credits used and remaining for the period. Per-method and per-chain breakdowns are at `/v0/usage/rpc/by-method` and `/v0/usage/rpc/by-chain` ŌĆö `by-method` also flags each method as `archive: true`/`false`, which maps to Chainstack's 2 RU per archive request (vs 1 RU standard), so you can convert precisely instead of by an average. If the user would rather not create a key, a **screenshot** of the **Usage & Billing** page works too ŌĆö read the figures straight from the image.
   - **Alchemy** ŌĆö the [Alchemy CLI](https://www.alchemy.com/docs/alchemy-cli) exposes usage programmatically. Have the user install it (`npm i -g @alchemy/cli`) and run `alchemy auth login` once ŌĆö a one-time browser sign-in to their own Alchemy account (there is no headless API-key path for `usage`). Then fetch their spend and compute-unit consumption:

     ```
     alchemy --json usage summary
     ```

     Read `data.totals.last30Days.usd` (30-day spend), `data.totals.last30Days.amount` (Compute Units), and `data.usageLimit` (their plan cap). For a per-network breakdown over time, use `alchemy --json usage timeseries --granularity day --metrics usd amount --group-by network --start-date <date>` ŌĆö note the time-series range is plan-limited (1 day on Free, 7 on Pay-as-you-go), whereas the `summary` figures above are not range-limited. If the user would rather not install the CLI, a **screenshot** of their dashboard **Usage** (Settings ŌåÆ Usage) and **Billing** page works too ŌĆö read the figures straight from the image.
   - **Other provider / no programmatic access** ŌĆö ask for a **screenshot** of their usage/billing dashboard, or their approximate monthly request volume and current spend.
6. Call `get_chainstack_pricing` for live Chainstack plans and the Request Unit (RU) model. The snapshot already includes the per-provider conversion multipliers (QuickNode API Credits ŌåÆ RU, Alchemy Compute Units ŌåÆ RU) and a worked migration example ŌĆö use those live values rather than restating numbers here. Convert the source usage to RU, fit it to the cheapest plan that covers it, and present the monthly cost beside their current bill.
7. Route on the estimate: high volume, dedicated-node needs, or Enterprise ŌåÆ `contact_chainstack` (or [@ChainstackGrowthBot](https://t.me/ChainstackGrowthBot)); self-serve plans ŌåÆ [console.chainstack.com](https://console.chainstack.com).

#### Dry run

If the user already has Chainstack nodes deployed (API key required):

8. Use `list_nodes` to get all deployed Chainstack endpoints.
9. For each RPC method found in the codebase, send a test request to the corresponding Chainstack endpoint to verify it works.
10. Optionally, if the codebase has RPS patterns (batch calls, concurrent requests, polling intervals), test at that throughput. Use `search_docs` to find additional information and recommended optimizations.
11. Report what works, what fails, and any rate limit or method support gaps.

### Plan

12. For each endpoint to migrate, lay out the specific replacement ŌĆö which existing Chainstack node covers it, or whether a new node needs to be created. Flag anything that needs the user's input (tier choice, project assignment, unsupported chains).

<Warning>
  STOP here. Present the plan and wait for the user to approve or adjust before making any changes. Do not proceed to implementation until the user explicitly confirms. Answer any follow-up questions ŌĆö use the MCP server tools to provide accurate, specific answers. For implementation issues, direct the user to [Chainstack support](https://support.chainstack.com/).
</Warning>

### Implement

13. Create any nodes needed via `create_node` (use global nodes for instant deployment).
14. Replace each endpoint with the corresponding Chainstack URL from `list_nodes`.
15. Summarize what changed.
