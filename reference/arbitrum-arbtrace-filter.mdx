Arbitrum API method that allows to retrieve internal transaction data by defining a specific filter object. This method is a powerful tool for gaining insight into the intricate operations of complex transactions, particularly those associated with smart contracts. It provides you with the capability to seamlessly access all internal transactions connected to a particular address within a predefined block range.

<Note>
Learn how to [deploy a node](/docs/debug-and-trace-apis#arbitrum) with the debug and trace API methods enabled.
</Note>

<Warning>
Blocks older than 22,207,815 were added to the chain before the Nitro migration and cannot be queried with Geth `debug_*` methods. Starting from block 22,207,815, Arbitrum migrated to Nitro which made Geth `debug_*` methods available for newer blocks.

Use the `arbtrace_filter` method for calling blocks prior to 22,207,815.
</Warning>

## Parameters

* `object` ‚Äî the filter object used for specifying the following filter criteria:
  + `fromBlock` ‚Äî (optional) specifies the starting block from which to retrieve data.
  + `toBlock` ‚Äî (optional) specifies the ending block until which data should be retrieved.
  + `fromaddress` ‚Äî (optional) array of addresses of the senders from which the transactions should be filtered.
  + `toaddress` ‚Äî (optional) array of addresses of the receivers to which the transactions should be filtered.
  + `after` ‚Äî (optional) specifies the offset trace number to start retrieving traces.
  + `count` ‚Äî (optional) specifies the number of traces to display in a batch.

## Response

* `array` ‚Äî represents the block traces, which consist of the following fields (note that all return types are hexadecimal representations unless otherwise stated):
* `action` ‚Äî represents the `ParityTrace` object, which provides details on the following:
  + `from` ‚Äî indicates the address of the sender.
  + `callType` ‚Äî specifies the type of method used, such as `call` or `delegatecall`.
  + `gas` ‚Äî represents the gas provided by the sender, encoded in hexadecimal.
  + `input` ‚Äî contains the data sent along with the transaction.
  + `to` ‚Äî specifies the address of the receiver.
  + `value` ‚Äî indicates the value sent with this transaction, encoded as an integer in hexadecimal.
* `blockHash` ‚Äî provides the hash of the block where this transaction was included.
* `blockNumber` ‚Äî represents the block number in which this transaction was included.
* `result` ‚Äî refers to the `ParityTraceResult` object, which includes the following information:
  + `gasUsed` ‚Äî represents the amount of gas used by this specific transaction alone.
  + `output` ‚Äî contains the value returned by the contract call. It includes the actual value sent by the `RETURN` method. If the `RETURN` method was not executed, the output is empty bytes.
* `subtraces` ‚Äî represents the traces of contract calls made by the transaction.
* `traceAddress` ‚Äî specifies the list of addresses where the call executed, including the address of the parents and the order of the current sub-call.
* `transactionHash` ‚Äî refers to the hash of the transaction.
* `transactionPosition` ‚Äî indicates the position of the transaction.
* `type` ‚Äî specifies the type of method used, such as `call` or `create`.

## Use case

The `arbtrace_filter` method on the Arbitrum blockchain allows you to retrieve detailed traces of smart contract execution. Think of traces as a step-by-step account of what happens when a smart contract is called or interacts with other contracts.

With this method, you can specify filter criteria, such as the range of blocks you're interested in. When you invoke the `arbtrace_filter` method, it returns traces that match your criteria.

Traces are useful for various purposes. For example, you can debug smart contract transactions by analyzing the sequence of function calls, state changes, and events emitted during execution. This helps you identify and fix issues in your contract's logic.

Traces also provide insights into gas consumption, allowing you to optimize gas usage in your contracts by identifying gas-intensive operations.
