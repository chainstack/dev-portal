The `getBlockTransactions` method retrieves transactions from a specific block in the TON blockchain. This method allows you to fetch transaction details for a given block, specified by its workchain, shard, and sequence number.

## Parameters

* `workchain` (integer, required) ‚Äî The ID of the workchain. Example: `-1` for the masterchain.
* `shard` (string, required) ‚Äî The ID of the shard. Example: `"-9223372036854775808"` for the masterchain.
* `seqno` (integer, required) ‚Äî The sequence number of the block. Example: `39064874`.
* `count` (integer, optional) ‚Äî The maximum number of transactions to return. Default: `40`.

## JSON-RPC

```shell Shell
curl -X POST \

  -H 'Content-Type: application/json' \
  -d '{

    "method": "getBlockTransactions",
    "params": {
      "workchain": -1,
      "shard": "-9223372036854775808",
      "seqno": 39064874,
      "count": 40
    }
  }'
```


## Response

* `id` (object) ‚Äî The identifier of the block, containing:
  + `workchain` (integer) ‚Äî The workchain ID of the block.
  + `shard` (string) ‚Äî The shard ID of the block.
  + `seqno` (integer) ‚Äî The sequence number of the block.
  + `root_hash` (string) ‚Äî The root hash of the block.
  + `file_hash` (string) ‚Äî The file hash of the block.
* `transactions` (array of objects) ‚Äî An array of transaction objects, each containing:
  + `account` (string) ‚Äî The account address involved in the transaction.
  + `hash` (string) ‚Äî The transaction hash.
  + `lt` (string) ‚Äî The logical time of the transaction.
* `incomplete` (boolean) ‚Äî Indicates if the returned list of transactions is incomplete.

## Use case

A possible use case for the `getBlockTransactions` method in TON is for blockchain explorers, analytical tools, or any application that needs to examine transactions within specific blocks. This method can be used to:

1. Display transaction details for a given block in a block explorer.
2. Analyze transaction patterns or volumes within specific blocks or shards.
3. Monitor specific accounts by tracking their transactions across blocks.
4. Implement custom indexing or caching mechanisms for transaction data.
